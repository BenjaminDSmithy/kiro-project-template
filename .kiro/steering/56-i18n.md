# Internationalisation (i18n) Standards

Patterns for multi-language support in Next.js App Router.

## Decision Tree: Server vs Client Translation

```text
1. Is the text rendered in a Server Component?  → Server-side translation (next-intl server)
2. Is the text in a Client Component?           → Client-side translation (useTranslations)
3. Is it metadata (title, description)?         → generateMetadata with locale param
4. Is it a date, number, or currency?           → Intl API (server or client)
5. Is it an error message from an API?          → Return error codes, translate on client
```

## Message File Structure

```text
messages/
├── en/
│   ├── common.json
│   ├── auth.json
│   └── orders.json
└── fr/
    ├── common.json
    ├── auth.json
    └── orders.json
```

## Message Key Conventions

| Rule                   | Correct                                         | Avoid                     |
| ---------------------- | ----------------------------------------------- | ------------------------- |
| Namespace by feature   | `auth.login.title`                              | `loginTitle`              |
| Descriptive keys       | `orders.empty.description`                      | `orders.msg1`             |
| Use ICU message format | `{count, plural, one {# item} other {# items}}` | Manual pluralisation      |
| Keep keys stable       | Never rename without migration                  | Renaming breaks fallbacks |

## Date, Number, and Currency Formatting

Always use the `Intl` API — never format manually:

```typescript
// Dates
new Intl.DateTimeFormat(locale, { dateStyle: "medium" }).format(date);

// Numbers
new Intl.NumberFormat(locale, { style: "currency", currency: "AUD" }).format(
  amount,
);

// Relative time
new Intl.RelativeTimeFormat(locale, { numeric: "auto" }).format(-1, "day");
```

## Rules

| Rule                                   | Rationale                                 |
| -------------------------------------- | ----------------------------------------- |
| Never hardcode user-facing strings     | All text must come from message files     |
| Store dates in UTC, format with locale | Consistent storage, localised display     |
| Use ICU message format for plurals     | Handles complex plural rules across langs |
| Provide fallback locale (en)           | Graceful degradation for missing keys     |
| Extract strings during development     | Don't defer i18n to "later"               |
| Keep message files flat where possible | Easier for translators to work with       |
